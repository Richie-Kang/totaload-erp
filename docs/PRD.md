# PRD: 중고차 수출 ERP — Hanaru AI ERP

> 이 문서는 `phases/0-mvp/step0` 에서 마스터 계획서(`ethereal-wondering-puzzle.md`) 1절을 옮겨 작성했다. 이후 모든 step 의 가드레일로 주입된다.

## 0. 맥락 (왜 만드는가)

중고차 수출 회사 **트리닉(TRYNIC Co., Ltd / 사업자 860-10-02700, 인천 미추홀구)** 의 사내 ERP "Hanaru AI ERP". 수출 차량은 출고 전 **자동차 말소등록**(자동차관리법 §13①, 자동차등록령 §31①, 자동차등록규칙 §37, 별지 제17호 서식)을 해야 한다. 현재는 직원이 자동차등록증을 보며 손으로 신청서 PDF 폼을 채우고, 과거 차량 정보를 찾을 때 파일 폴더를 뒤진다 → 느리고 오타가 잦다.

MVP 가 해결하는 것: (1) 자동차등록증 사진 한 장 → 말소등록 신청서 PDF 자동 작성(검수·수정 가능), (2) 차량번호/차대번호 일부로 과거 차량과 서류를 즉시 검색. 다른 직원들도 배포된 URL 로 사용.

스택(사용자 지정): React(프론트) / Node.js(API) / Python(OCR·PDF) / PostgreSQL. OCR 은 백엔드가 ocr-service 를 통해 로컬 `codex` CLI 를 호출해 수행(별도 OCR API 키 없이 사용자의 ChatGPT/Codex 구독 사용). 배포는 PaaS(Render Blueprint).

### 입력 자료 (리포지토리에 존재)
- 폼 템플릿: `assets/malso_application_template.pdf` (2페이지, AcroForm 텍스트 필드 12개; 1페이지=말소등록 신청서, 2페이지=Commercial Invoice 트리닉 양식).
- 샘플 자동차등록증 이미지: `assets/samples/20251203_자동차등록증.jpg`, `assets/samples/자동차등록증_레이.jpg`.

## 1. 목표
자동차등록증 정보를 OCR 로 읽어 말소등록 신청서 PDF 를 자동 작성하고, 차량·서류를 차량번호/차대번호로 즉시 검색할 수 있게 하여 트리닉의 수출 차량 말소 업무 시간을 줄이고 오타를 없앤다.

## 2. 사용자
- **수출 담당 직원** (주 사용자): 차량을 인수받아 등록증을 받고 말소 신청서를 만든다. 하루 수 건~수십 건.
- **사무/관리 직원**: 과거 차량·서류를 조회한다.
- 모두 사내 인원. 외부 공개 없음. (MVP 에선 로그인 없이 누구나 접근 — 아래 보안 절 참조.)

## 3. 핵심 기능

### 기능 1 — 말소 입력 (사이드바 첫 항목)
1. 자동차등록증 이미지(JPG/PNG/PDF) 업로드(드래그&드롭 또는 파일 선택).
2. 시스템이 이미지를 저장하고 OCR(=codex CLI)을 실행 → 등록증의 필드들을 추출.
3. 추출 결과를 **편집 가능한 폼**으로 보여준다(업로드한 이미지 미리보기를 옆에 나란히). 추출 못 한 필드는 빈칸 + 표시. 모든 필드 수정 가능.
4. 사용자가 검수·수정 → "저장"으로 차량 레코드 갱신(자동 저장).
5. "말소등록 신청서 PDF 생성" → 현재 폼값으로 템플릿 PDF 의 12개 필드를 채운 PDF 를 만들어 다운로드/미리보기. 생성된 PDF 도 차량에 첨부 저장.
6. (선택) 다시 수정 후 PDF 재생성 가능.

### 기능 2 — 말소 검색 (사이드바 둘째 항목)
1. 검색창에 차량번호 또는 차대번호의 **일부** 입력(부분일치, 대소문자/공백 무시).
2. 일치 차량 목록(차량번호 · 차대번호 · 차명 · 상태(작성중/완료) · 등록일).
3. 행 선택 → 상세: 모든 필드 표 + 첨부된 자동차등록증 이미지(미리보기/다운로드) + 생성된 말소등록 신청서 PDF(미리보기/다운로드). 상세에서 바로 수정·PDF 재생성 가능(=말소 입력 폼 재사용).

## 4. 채워야 하는 PDF 폼필드와 자동차등록증 매핑 (12개)

| PDF 필드명 | 위치 | 소스(자동차등록증) | 처리 규칙 |
|---|---|---|---|
| `owner_name` | 1p 소유자 성명(명칭) | 소유자 성명/명칭 | 그대로 |
| `owner_ssn` | 1p 주민(법인)등록번호 | 주민등록번호 또는 법인등록번호 | 그대로 (민감정보 — 7절 참조) |
| `owner_address` | 1p 주소 | 소유자 주소(없으면 사용본거지) | 줄바꿈 제거, 공백 정리 |
| `vehicle_reg_no` | 1p 자동차등록번호 | 자동차등록번호 | 예: `123가4567` / `12가3456` |
| `vehicle_vin_1` | 1p 차대번호 | 차대번호(VIN) | 영문 대문자/숫자, 공백 제거 |
| `vehicle_vin_2` | 2p Descriptions of goods | 차대번호(VIN) | `vin_1` 과 동일 값 |
| `vehicle_model` | 2p 품명 | 차명 | 그대로 |
| `vehicle_year ` | 2p (필드명 끝에 **공백 1개** — 정확히 `"vehicle_year "`) | 형식 및 연식 | 가능하면 4자리 연도, 아니면 원문 |
| `vehicle_mileage` | 1p 주행거리 km | 주행거리(앞면에 없을 수 있음) | 없으면 null → 사용자 수동 입력. 숫자만(콤마 제거) |
| `vehicle_weight_1` | 2p WEIGHT KG | 차량중량(공차중량) kg | 숫자만 |
| `vehicle_weight_2` | 2p 총합계 KG | 차량총중량 kg (없으면 `weight_1` 과 동일) | 숫자만 |
| `current_date` | 신청/인보이스 날짜 | — | 기본값 = 오늘(`YYYY년 M월 D일`), 사용자 수정 가능 |

- 폼의 "말소등록의 원인" 체크박스는 템플릿에 `[V]수출 예정`, `[V]말소사실증명서 발급 필요` 가 이미 표시돼 있고 폼필드가 없다 → 건드리지 않는다.
- 2페이지 인보이스의 회사 정보(트리닉 상호/주소/사업자번호)도 템플릿에 인쇄돼 있어 채울 필요 없다. 인보이스의 단가/금액 등은 폼필드가 아니므로 범위 밖.

## 5. 사용자 시나리오 (수용 시나리오)
- **S1 정상**: 직원이 등록증 사진 업로드 → 10~40초 후 폼이 추출값으로 채워짐 → 주행거리만 손으로 입력 → "PDF 생성" → 모든 칸이 채워진 신청서 PDF 다운로드 → 인쇄.
- **S2 OCR 부분 실패**: 일부 필드만 추출됨 → 빈칸은 노란 테두리로 표시 → 직원이 등록증 이미지를 보며 채움 → PDF 생성. 시스템이 막지 않음.
- **S3 OCR 전면 실패(codex 오류)**: 명확한 에러 메시지 + "직접 입력" 폼(모든 필드 빈칸)으로 폴백 → 직원이 전부 입력 → PDF 생성 가능. 업로드 이미지·원시 응답은 보존.
- **S4 검색**: "3456" 입력 → 차대번호/차량번호에 3456 포함 차량 목록 → 한 건 선택 → 등록증 원본과 생성 신청서 확인 → 필요 시 수정·재생성.
- **S5 재업로드(중복)**: 같은 차(같은 VIN) 등록증을 또 업로드 → 새 레코드를 만들지 않고 기존 레코드를 열어 편집(상태/첨부 유지). VIN 추출 실패 시엔 새 레코드 생성.

## 6. 비기능 요구
- **성능**: OCR 호출은 동기 처리, 타임아웃 90초, 로딩 표시. 검색 응답 < 1초(인덱스). 동시 사용자 소수(<10).
- **신뢰성**: codex/OCR 실패가 업무를 막지 않는다(항상 수동 폴백). PDF 생성은 OCR 없이도 가능.
- **데이터 보존**: 업로드 이미지·생성 PDF 는 영구 디스크에 보관. DB 는 Render 관리형 Postgres. (백업은 운영 과제 — MVP 범위 밖이나 README 에 명시.)
- **이식성**: 로컬은 `docker compose up` 한 줄, 배포는 Render Blueprint 한 번.
- **국제화**: UI 한국어. 등록증/신청서 한국어. VIN/숫자만 ASCII.

## 7. 보안·프라이버시 (MVP 수준이지만 명시)
- **주민등록번호는 민감정보**. MVP 에선 로그인이 없으므로: ① 목록/검색 결과에는 SSN 을 노출하지 않는다(상세 화면에서만, 그것도 `8601**-*******` 마스킹 후 "보기" 토글 시 평문). ② DB 컬럼에 평문 저장하되 README 에 위험 명시(향후 컬럼 암호화·접근제어 과제). ③ PDF 생성 시에만 평문 사용.
- 파일 접근은 `GET /api/files/:docId` — DB 의 docId 로만 조회, 사용자 입력 경로/파일명 절대 사용 안 함(경로 traversal 차단). 파일은 UUID 파일명으로 저장.
- 검색은 전부 parameterized query(SQL injection 차단).
- 업로드 검증: 확장자·MIME·매직바이트 확인, 최대 20MB, 이미지/PDF 만 허용.
- CORS: 프론트 오리진만 허용.
- 비밀값(`CODEX_AUTH_JSON`, `DATABASE_URL`)은 env 로만, 리포에 커밋 금지(`.gitignore` 에 `.env`, `~/.codex/auth.json` 류 포함).

## 8. MVP 제외 사항
- 로그인 / 사용자 계정 / 권한 / 감사 로그.
- 차량 재고관리, 매입·매출, 정산, 환율, 인보이스 자동 산출(금액/단가).
- 수출 신고(관세청), 선적, 운송, B/L.
- 통계·대시보드·리포트.
- 모바일 전용 UI(반응형은 best-effort).
- 다국어, 알림(이메일/카톡).
- OCR 결과 정확도 보장·재학습.

## 9. 성공 지표
- 등록증 1장 → 인쇄 가능한 신청서 PDF 까지 1분 이내(정상 케이스).
- 정상 등록증에서 12필드 중 9개 이상 자동 채움(주행거리·연식 제외 시 대부분).
- 검색으로 과거 차량을 10초 안에 찾고 등록증·신청서 열람.

## 10. 미해결/사용자 확인 필요 (실행 중 기본값으로 진행)
- `DELETE /api/malso/:id` 를 MVP 에 둘지(잘못 만든 레코드 정리용) — 기본 **미포함**.
- 차량별 자유 메모/비고 필드(`note text`) 추가 여부 — 기본 **추가**(PDF 와 무관, 상세/검수 화면 하단 텍스트영역, 검색 대상 아님).
- `current_date` 표기 — 기본 `YYYY년 M월 D일`(사용자 수정 가능).
- `vehicle_weight_2`(인보이스 총합계 KG) — 기본 **차량총중량(없으면 차량중량 복제)**.
- 입력 PDF 멀티페이지 시 1페이지만 OCR(ADR-010) — 수용 가정.
- OCR 진행 중에도 빈 폼에 선입력 가능, 결과는 빈 칸만 채움 — 확정.
